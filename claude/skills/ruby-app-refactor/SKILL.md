# Ruby App Refactor

Improve one poorly-graded Ruby file per run. Keep the change small enough to review in a single sitting. This is a standalone, low-stakes improvement — not a fix for a specific bug or ticket.

## Preconditions

- Project uses Ruby and has rubycritic available (via `bundle exec rubycritic` or the global gem).
- The `wt` (worktrunk) CLI is installed — all work happens in a worktree, not the main checkout.

## Workflow

### 1. Create a Worktree

Use `wt` to create a dedicated worktree with a predictable, dated branch name so repeated runs don't collide and PRs are easy to find later:

```bash
wt switch --create refactor/automated-$(date +%Y-%m-%d)
```

This creates the worktree, checks out the new branch, and switches you into it. All subsequent steps run inside the worktree. If `wt` is not installed or fails, stop and tell the user.

### 2. Run RubyCritic

Scope to the main source directories (typically `app/` and `lib/`; adjust to the project). Exclude migrations, vendored code, and generated files:

```bash
bundle exec rubycritic --no-browser -f console app lib
```

Fall back to the global gem if the project doesn't bundle it:

```bash
rubycritic --no-browser -f console app lib
```

### 3. Pick One File

Prefer files in this order: F → D → C. Skip A and B.

Favor implementation files in `app/` and `lib/`. Refactoring a spec is allowed but should be the exception — only do it if the spec itself is clearly the problem.

Skip candidates that are likely to cause trouble:
- Migrations, seeds, vendored code, generated code
- Files with recent commits from other contributors or that appear in open PRs — check `git log -- <file>` and `gh pr list --search "<file>"`
- Files that were refactored in a recent run — check `gh pr list --state all --search "refactor/automated in:title" --limit 10`

If nothing looks safe, report that and stop. Skipping a run is fine.

### 4. Assess Test Coverage

Find the corresponding spec file (e.g., `spec/.../<file>_spec.rb`).

- **No specs:** Add meaningful coverage for the behavior you're about to touch before refactoring. You need a safety net.
- **Sparse specs:** Fill in coverage for the code paths you plan to change.
- **Good coverage:** Proceed.

Run the relevant tests and confirm they pass before changing any production code. A green baseline is required.

### 5. Refactor

Follow the Test-Driven Development rules. Make one focused change at a time and keep tests green between steps.

Target the specific smells RubyCritic flagged (Reek warnings, Flog complexity, Flay duplication). Common useful moves:
- Extract method for long methods or high-complexity methods
- Rename poorly-named methods or variables
- Remove dead code
- Consolidate duplicated logic

Resist the urge to clean up unrelated code in the same PR. Keep the diff tight so the reviewer can evaluate it quickly.

### 6. Verify Improvement

Re-run RubyCritic against the touched files:

```bash
bundle exec rubycritic --no-browser -f console <file> <spec_file>
```

Confirm the grade improved. If it didn't:
- Try a different angle on the same file, or
- Abandon this file, reset the branch, and pick another

### 7. Run Project Checks

Run the project's full check command (typically `chp` or equivalent) and confirm everything is green before opening a PR.

### 8. Create a Draft PR

Invoke the `/create-pr` skill to open the PR in the current repository. Provide this context to that skill:

- **Title:** `Automated Refactor by Claude: <file name>` (Include 1 or 2 parent directories if using just the base file name would be ambiguous.)
- **No ticket** — this is an automated code health pass, not a ticketed change. Skip the "Contributes to..." line in the description.
- **Body highlights:**
  - Which smells RubyCritic flagged (Reek warnings, Flog score, duplication) and how the refactor addresses them
  - Before/after grade and relevant metrics
  - Note that the PR was generated by the `/ruby-app-refactor` skill
- **Keep it a draft.** The user reviews and promotes it to ready when appropriate.

### 9. Deploy to Stage

Follow the Stage Deployment guidelines to merge the branch into `stage`, run checks, push, and monitor. This lets the user test the refactor in a running environment before merging the PR.

### 10. Report

After the PR is open and stage is deployed, tell the user:
- File that was refactored
- Grade before → after
- Key smells addressed
- PR URL
- Stage deploy status
